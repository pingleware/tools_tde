     Proposals to change syntax of internal commands used by CB IDE to generate success build of targets.

CB use an internal format of different commands to generate "target" dependant of kinds of configured 
compilers with objective to respect "correct" syntax to achieve build process. This syntax, describe 
above with different kind of compilers, use different "symbol name" named "command macro" like "$compiler" 
to design executable to compile or like "$file" to design input file of compilers's tool, or "$link_objects" 
to design list of object files needed for linkage, etc...  

Example with compiler GNU C/C++ (and all of derivatives of GCC integrated into package or development's environment)

	1) command : compile single file to object file 
		$compiler $options $includes -c $file -o $object
	2) command : generate dependencies for file
		$compiler -MM $options -MF $dep_object -MT $object $includes $file
	3) command : compile Win32 resource file 
		$rescomp $res_includes $res_options -J rc -O coff -i $file -o $resource_output
	4) command : link object files to executable 				(or GUI executable on Win32 systems)
		$linker $libdirs -o $exe_output $link_objects $link_resobjects $link_options $libs -mwindows
	5) command : link object files to console executable
		$linker $libdirs -o $exe_output $link_objects $link_resobjects $link_options $libs
	6) command : link object files to dynamic library
		$linker -shared -Wl,--output-def=$def_output -Wl,--out-implib=$static_output -Wl,--dll $libdirs  \
				$link_objects $link_resobjects -o $exe_output $link_options $libs
	7) command : link object files to static library
		cmd /c if exist $static_output del $TO_WINDOWS_PATH{$static_output}
		$lib_linker -r -s $static_output $link_objects
	8) command : link object files to native executable 
		$linker $libdirs -o $exe_output $link_objects $link_resobjects $link_options $libs -Wl,--subsystem,native
		
Remark 1 : --subsystem,native indicate choice to generate Kernel mode drivers for Windows NT (and superior). 
		This option is normally reserved for Windows system components.
		
Remark 2 : To generate DLL on Win32 with GCC, developers of CB had chooseen to add two supplemental options 
		to generate automatically the import library and the definition file of DLL (shared library). 
		Very good choice.
		And these options appear into CB interface during creation of DLL project, but, it seem that they are
		choose or not by user of IDE. Strange, with GCC this choice seem to be fix and not optional. True ?
		
Remark 3 : Why presence of command before creation of static library like 
		cmd /c if exist $static_output del $TO_WINDOWS_PATH{$static_output} ?
		Clean process don't suppress file with extension ".lib" or ".a" ? Strange, why not ?
		
With GNU C/C+, these syntax are clearly identified, because this compiler is very common on multiple platforms.
And during his startup, CB can detect automatically presence of this compiler on Win32 platforms.
Good, but with C/C++ compilers unrecognized by CB, or with syntax not aligned with last version or evolution of 
compilers installed on your platform, the going gets tough ... 

Then, after list of "major options" of different tools integrated into all kinds of "free" compilers C/C++ 
available on Win32 plateforms, and also to distinguish target generation beetween 32 or 64 bit, it seem mandatory  
to precise if CB respect or not good syntax to build targets, and contribute to suggest evolutions in order to 
ameliorate this IDE.  

Focus are do to success build DLL target on Win32 systems, but, if possible, to generate in parallel "import library"
too and also used, if possible, and definition file in input to generation to permit manage of alias of exported 
symbols of DLL. All a program !

Principals format's syntax used by this compiler are :

	1) to compile C or C++ source files (space separated command)		 
		bcc32 [ options ] file[s]		(if compile only, generate .obj file, otion "-c" mandatory)

echo "************      Generation of DLL (only standard arithmetic operations on int and on float) in two passes      ***************"
REM Format general of bcc32 command : BCC32 [ options ] file[s]  
REM Options used with Borland compiler C/C++ 32 bits version 5.5.1 :
REM		-c 								-> compile only, not call of linker
REM		-q 								-> set to quiet mode (don't see copyrigth information)		
REM 	-WDR							-> set generation to Windows DLL -WD (mandatory or not in two passes ?) and -WR to indicate that target uses the dynamic RTL
REM 	-RT-							-> disable runtime type id (RTTI) (default is -RT)
REM 	-x- 							-> disable exception handling 
REM 	-Dxxxxxxxxxx 	 				-> define a variable xxxxxxxxxx used by BC55 precompiler
bcc32 -c -q -WDR -RT- -x- -DBUILD_DLL -DNDEBUG src\dll_core.c 
REM Format general of ilink32 command : 
REM     ILINK32 [@<respfile>][<options>] <startup> <myobjs>, [<exefile>],[<mapfile>], [<libraries>], [<deffile>], [<resfiles>]  (comma separated list of files used by linker)
REM 		<startup> can take values like : c0w32 for windows application (GUI), c0x32 for console application, or c0d32 for DLL application
REM Options used with linker of Borland C/C++ compiler 32 bits version 5.5.1 :
REM 	-Tpd							-> set the linker to generate DLL
REM 	-aa 							-> set the linker to generate Windows 32 application (NT system or superior, like Windows 7, 8, 10 or 11)
REM 	-x								-> enable exception handling
REM 	/Gi    							-> option -Gi tell linker to generate .lib file, but nothing done. Change option to /Gi active generation of lib file.
REM 	c0d32							-> Startup provide by Borland/Inprise for DLL (another startup obj, can be c0w32 for windows application (GUI), or c0x32 for console application)
REM  At the end of this command's line of linker, you must/can indicate def file, option not possible with bcc32 command line during one pass generation (compil + linkage) !!!!! 
echo "*****************    Edition des liens (linkage) de la DLL.      ***************"
ilink32 -Tpd -aa -x -q /Gi c0d32 dll_core.obj,dll_core.dll,,import32 cw32i, src\dll_core.def
REM    IMPLIB not mandatory, because option -Gi used by linker don't generate library file, but option /Gi do. Good surprise !!!
REM implib -c dll_core.lib dll_core.dll
echo.  ***************** 	       Listage des symboles exportes de la DLL 32 bits				*****************
REM     Use of "tdump" utility" to extract export symbols of dll. It's fine : find symbols/functions with "_", but although without this prefix (thank's "def file" with alias)
tdump -q -ee dll_core.dll 

All previous internals commands used by CB are adapted to Borland C/C++ like this :

	1) command : compile single file to object file 
		$compiler -q $options $includes -o$object -c $file
	2) command : generate dependencies for file
		...     empty field     ...     no equivalent command available with BC55 that with GCC ...
	3) command : compile Win32 resource file 
		$rescomp -32 -fo$resource_output $res_includes $res_options $file
	4) command : link object files to executable 				(or GUI executable on Win32 systems)
		$linker -q -aa  $link_options $libdirs c0w32 $link_objects,$exe_output,,$libs,,$link_resobjects
	5) command : link object files to console executable
		$linker -q -ap  $link_options $libdirs c0x32 $link_objects,$exe_output,,$libs,,$link_resobjects
	6) command : link object files to dynamic library
		$linker -q $libdirs -Tpd $link_options $link_objects, $exe_output,, $libs,, $link_resobjects
	7) command : link object files to static library
		$lib_linker -r -s $static_output $link_objects
	8) command : link object files to native executable 
		$linker -q -ap  $link_options $libdirs c0x32 $link_objects,$exe_output,,$libs,,$link_resobjects

Remark 1 : Last choice is redundant with command "link object files to console executable". Strange ... 
		BC55 is able to construct "system" DLL on Windows NT, you must just add option "/Gz" during link :
			/Gz 	Calculate checksum of target and insert result in PE header. 
					Used for NT Kernel mode drivers and system DLLs. Default is Off.

Remark 2 : Neither options added to construct import library or definition file during build of DLL.
		Shame, it's possible to generate import library during this build by add "/Gi" option (Warning
		"-Gi" don't work). And to generate definition file, BC55 provided tool to generate it "impdef" :
			
			Borland Impdef Version 3.0.22 Copyright (c) 1991, 2000 Inprise Corporation
			Syntax: IMPDEF [options] destname[.def] srcname[.dll]
			Options:
					-a     Add '_' alias for MS flavor cdecl functions
					-h     Emit hints
		
		It can be added by just this syntax : cmd /c impdef $exe_dir\$exe_name.def $exe_output in "Post
		operations" after build DLL.

Remark 3 : For third internal command (compile resource file), the option "-32" is the default. Why force 
		it ?
		
Remark 4 : In real "world", it can be useful to use an definition file during linkage in "input" to authorize
		alias (by example) of functions used into DLL (by example, alias to suppress prefix "_" with "cdecl" 
		calling convention). With this, you can call function directly by it's name with script Python by example.
		Syntax of BCC5 linker permit it, but this option is not implemented into internals commands of CB.

B) With CLANG C/C++ compiler version 19.1.n  (compiler not detected automaticaly by CB ...)

CLANG C/C++ compiler is modern implementation of C/C++ compiler based on new concepts oriented on "portability".
It has been created after constat that GNU C/C++ compiler is become "a very big monolithic program", hard to update,
hard to maintain (many platforms ARM/ARM64, PowerPC, and many languages supported Fortran, Ada, Objective C/C++, 
etc...,) same if this "old" compiler has been essential because very used during long time, many years, and free.
Don't forget that Kernel Linux are generated by this ... same if Rust begin to be used also to build this kernel.

But, CLANG C/C++ compiler is unsuffisant to generate operational target on platforms like Windows (or Linux).
It must be leaned with full development package to operate efficiently, like with GCC/MinGW (too it !) or like
MSVC, last packages provided libraries and include files, and some tools, mandatory to success building.

Warning, into CB, if you want configure this compiler, you must identify all binary's directories, all library's
directory and all include's directories used by CLANG C/C++ but also used by leaned package choosen.

Then to configure into CB IDE an configuration adequate like this, after select main menu "Settings" then select 
submenu "Compilers..." and after select "GNU GCC Compiler" in list choice on top of open box opened if you want
to use pair CLANG+MinGW to build target, or select "Microsoft Visual C++ 2022" (I rename "Microsoft Visual C++ 2017"
to adapt with last Visual Studio 2022 Community (free) downloadable from site Microsoft) if you want to use 
CLANG+MSVC to build targets

1) First choice : use of pair CLANG+MinGW
			
After select "GNU GCC Compiler", you must rename it with "CLANG MSYS2 W64" (by example, if you choose CLANG leaned
with MinGW64 integrated into MSYS2 64 bit installation). You can choose many implementation of MinGW, if you have

********************************************************************************************************************** 
What functions are exported when build dll using Mingw gcc on Win32 systems ?
Date : June 27, 2024	

If you build a dll using Mingw g++ like:

g++ -shared -o myprogrammingnotes.dll myprogrammingnotes.cpp

If __declspec(dllexport) is not present in the source file, all functions except the entry point function(DllMain) in 
myprogrammingnotes.cpp will be exported by name.

If __declspec(dllexport) is present in the source file, the functions decorated by __declspec(dllexport) will be 
exported by name, while those functions without __declspec(dllexport) will not be exported. 
You can decorate DllMain with __declspec(dllexport) to export it, too.

You can also use .def to specify which functions are to be exported. 
Then, the functions decorated by __declspec(dllexport) plus the functions specified in .def are exported.

    //mydll.def
    EXPORTS  
       Sum   @2  

how to use the .def file with Mingw g++?

g++ -shared -o mydll.dll mydll.cpp mydll.def

If you want to export a function by ordinal rather than by name, you can use the NONAME attribute in .def file like

    //mydll.def
    EXPORTS  
       Sum   @2  NONAME

You can specify the ordinal for a function in .def (the number after @). The ordinal must be 1 or bigger. The least 
number will become the ordinal base in the image. 
The ordinals may be not continuous, in which case, NULL entries will occur in the export address table of the dll.

Internal commands used by CB with Pelles C compiler are described after (same as MSVC, strange not ?  -) ):

	1) command : compile single file to object file 
		$compiler /nologo $options $includes /c $file /Fo$object
	2) command : generate dependencies for file
		...     empty field     ...     no equivalent command available with MSVC compilers that with GCC ...
	3) command : compile Win32 resource file          
		$rescomp $res_includes $res_options -fo$resource_output $file
	4) command : link object files to executable 				(or GUI executable on Win32 systems)
		$linker /nologo /subsystem:windows $libdirs /out:$exe_output $libs $link_objects $link_resobjects $link_options
	5) command : link object files to console executable
		$linker /nologo /subsystem:windows $libdirs /out:$exe_output $libs $link_objects $link_resobjects $link_options
	6) command : link object files to dynamic library
		$linker /dll /nologo $libdirs /out:$exe_output $libs $link_objects $link_resobjects $link_options
	7) command : link object files to static library
		$lib_linker /lib /nologo $libdirs /out:$static_output $libs $link_objects $link_resobjects $link_options
	8) command : link object files to native executable 
		$linker /nologo /subsystem:native $libdirs /out:$exe_output $libs $link_objects $link_resobjects $link_options

J) With "Tiny C" compiler (32 and 64 bit).

Normally, after decompress two archive files to install Tiny C Compiler in directory C:\tcc proposed by default, CB 
can detect it, and it's identified by "Tiny C Compiler". 

But, if you want use 32 bit or 64 bit configuration of this compiler, you must access into IDE CB at menu "Settings" 
and submenu "Compilers" to select "Tiny C Compiler", and rename it, by example "Tiny C Compiler X32"

An after, you must position "Compiler's installation directory" to "C:\tcc" in subtab "Toolchain executables".
I recommand to add this path into "Additional Paths" to be sure to give access at tools into it, and not in "\bin" subdir.

This directory is the same beetween 32 or 64 bits, but the rest of configuration is different in 32 or 64 bits.
You must select list of tools like this (in tab "Program Files") :
	C Compiler : 				tcc.exe
	C++ Compiler :				
	Linker for dynamic libs :	tcc.exe
	Linker for statics libs :	tcc.exe 
	Debugger :					GDB/CDB 		(choice by default, because TCC is compatible with version GDB)
	Resource compiler :			windres.exe 	(tcc is compatible with this resource compiler provided by GNU)
	Make program : 				mingw32-make.exe 	(tcc don't provided this tool, but GNU with MinGW32/64 make tool)

For compiler, search directories of "include files", "lib files" and "Resource include files" are :
		C:\tcc\include
		C:\tcc\lib
		C:\mingw64\x86_64-w64-mingw32\include		(Strange ? Because "resource compiler" is "windres.exe" ?)
									(Why not use "C:\tcc\include" ?)
									
Into subtab "Compiler Settings" and "Linker Settings", you must add respectively in subtab "Other compiler options" 
and zone "Other linker options", same option "-m32" to force use of i386 toolchain during build.

And, to distinguish version 64 bit into CB, you must first copy "Tiny C Compiler X32" and rename it in "Tiny C Compiler X64".

To terminate, you must "force" an option in tab "Compiler Settings" and in subtab "Other compiler options" to "-m64", and 
add the same option in zone "Other linker options".

Syntax of  Tine C compiler "tcc" is very simple, to resume principal and useful options of command "tcc", you can use by 
example :

	"-m32" 				Generate i386 32 bit object or executable.
	"-m64" 				Generate x64 64 bit object or executable (default on X64 architecture ?)
	"-c"                Compile and assemble, but do not link.
	"-g" 				Enable debug information.
	"-D<macro[=value]>" Define macro to preprocessor, and optionnally attribute a value.
	"-o <file>"         Place the output into <file>
	"-I<directory>"		Add directory to search include files.
	"-L<directory>"	    Add directory to search library files (warning, "tcc" don't use import library file but "import" definition file only)
	'-lxxxxx" 			Use xxxxx "library" (not, it's "import" definition file) needed to build target 
								(ex : if opengl32.def exist into subdir "\lib" of tcc, you can use opengl in your code with "lopengl32") 
	"-shared" 			To build shared library (DLL on Win32 systems)
	"-w"          		Disable all warnings

And this same tool must be used to :
  create static library  	: 	tcc -ar [rcsv] lib.a files
  create def file 			: 	tcc -impdef lib.dll [-v] [-o lib.def]       			(like with "opengl32.dll" provided by W11 by example)

You can see after how to generate 64 bit DLL with this compiler into console commeand windows (CMD) :

echo. ******************          Compilation de la DLL en mode 64 bits        *******************
REM     Options used by Tiny C Compiler 64 bit
REM 		-m64      					Select x86_64 toolchain (64 bit)
REM 		-c       					Compile only, stop just before linkage
REM 		-Dxxxx						Define variable xxxxx used by preprocessor 
REM 		-o xxxxx 					Define output file generated by Tiny C Compiler, here obj file
tcc -m64 -c -DBUILD_DLL -DNDEBUG src\dll_core.c -o dll_core64.obj
echo. *****************                Edition des liens .ie. linkage de la DLL.        			***************
REM     Options used by linker of Tiny C Compiler  
REM 		-m64      					Select x86_64 toolchain (64 bit) to linkage
REM 		-shared     				Select "shared" generation, here to build DLL on Win32 systems
REM 		-o xxxxx 					Define output file generated by Tiny C Compiler, here dll file
tcc -m64 -shared dll_core64.obj -o dll_core64.dll
REM 	Tiny C compiler can also be used to construct static library file ("tcc -ar ...") or create "import" definition file (the last to be used after by test programs)
REM 		-m64      					Select x86_64 toolchain (64 bit) to generate "import" definition file 
REM 		-impdef     				Select generation of "import" definition file (TCC don't never use an import library file like others compilers ... but def file)
echo. ************    Dump des symboles exportes avec l option impdef puis type file genere     **************
tcc -m64 -impdef dll_core64.dll -o dll_core64.def
type dll_core64.def
REM 			You can generate version 32 bit of this DLL by exchange "-m64" by "-m32" in this command file

Internal commands used by CB with Tiny C compiler are described after :

	1) command : compile single file to object file 
		$compiler $options $includes -c $file -o $object
	2) command : generate dependencies for file
		...     empty field     ...    
	3) command : compile Win32 resource file          
		...     empty field     ...  
	4) command : link object files to executable 				(or GUI executable on Win32 systems)
		$linker $libdirs -o $exe_output $link_objects $link_resobjects $link_options $libs -mwindows
	5) command : link object files to console executable
		$linker $libdirs -o $exe_output $link_objects $link_resobjects $link_options $libs

						$link_resobjects -o $exe_output $link_options $libs
	7) command : link object files to static library
		...     empty field     ...  
	8) command : link object files to native executable 
		$linker $libdirs -o $exe_output $link_objects $link_resobjects $link_options $libs
